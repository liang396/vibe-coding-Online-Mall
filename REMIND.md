# 二手商城 Vibe Coding 全栈开发提醒

本文档是这个仓库的开发提醒文档。它不是泛泛的 AI 编程教程，而是结合当前项目的真实开发过程，总结一套适合本仓库的 Vibe Coding 全栈流程。

参考资料：
- 鱼皮《Vibe Coding 项目开发流程》提出了 5 步主线：Research、PRD、Tech Design、AGENTS、Build。
- 本仓库实际开发中又验证了两个关键点：先跑通核心链路，再逐步收紧规则；每次修改都要限定范围，否则很容易把别的功能带坏。

## 1. 先研究，不要一上来就让 AI 写代码

这个项目一开始的目标并不只是“做个二手商城”，而是要明确：
- 一个账号既能当买家也能当卖家，并且能切换身份
- 商品、购物车、订单、评论必须串成完整闭环
- 订单状态不能任意改，必须符合买家和卖家的实际流程

在这个阶段，先整理：
- 目标用户是谁
- 核心业务链路是什么
- 不做什么
- 已有竞品怎么处理登录、商品、订单、身份切换

建议把结果落到 `RESEARCH.md`。哪怕很短，也比没有强。

## 2. 把需求写成 PRD，先定边界再开发

本项目已经有 [PRD.md](/D:/二手/PRD.md:1)，但后续开发说明：PRD 不能只列功能名，必须写清楚业务状态和角色边界。

以本项目为例，PRD 至少要写清楚：
- 登录支持“用户名或邮箱”
- 用户默认具备买家和卖家能力，可切换当前身份
- 商品分类面向用户展示中文类型，不直接暴露分类 ID
- 订单完整流转为：待付款 -> 待发货 -> 待收货 -> 已完成
- 买家只能支付、确认收货；卖家只能发货；不能自由改任意状态
- 商品列表每页 10 条，避免页面失控

如果 PRD 只写“有订单功能”，AI 很容易生成一个任何人都能随便改状态的伪流程。

## 3. 写技术设计，尤其要写清配置覆盖和数据流

这个项目的真实技术栈：
- 后端：Spring Boot 3 + Spring Security + MyBatis + Redis + MySQL
- 前端：React + Vite + Axios
- 目录：后端在 `src/main/java/com/project`，前端在 `frontend/src`

这个阶段最重要的不是堆技术名，而是写清楚“系统怎么跑”：
- 配置有两份：根目录 [application.yml](/D:/二手/application.yml:1) 和 [src/main/resources/application.yml](/D:/二手/src/main/resources/application.yml:1)
- 根目录配置可能覆盖类路径配置，这会直接影响数据库初始化、Redis 连接和启动行为
- 数据库初始化脚本在 [src/main/resources/db/schema.sql](/D:/二手/src/main/resources/db/schema.sql:1) 和 [src/main/resources/db/data.sql](/D:/二手/src/main/resources/db/data.sql:1)
- 前端鉴权依赖 token 写入、本地存储和请求拦截器，关键逻辑在 [frontend/src/context/AuthContext.jsx](/D:/二手/frontend/src/context/AuthContext.jsx:1)

这一部分如果不写清，后面就会出现“明明改了配置但系统没生效”“数据库没建表”“登录后马上 403”这种问题。

## 4. 给 AI 明确规则，最好落成 AGENTS.md

鱼皮文档强调要有 AI 指令文件，这在本项目尤其重要。因为这个项目同时有：
- Java 后端
- React 前端
- MySQL 初始化脚本
- Redis 登录态
- 中英文界面文案

建议在后续补一份 `AGENTS.md`，至少写明：
- 开发环境是 Windows + PowerShell
- 修改前先读相关链路代码，不要猜
- 只改用户指定范围，不顺手重构无关模块
- 先修根因，再改提示文案
- 涉及登录、订单、角色切换时，必须同时检查前端请求字段、后端 DTO、权限校验、数据库结构
- 提交前检查有没有把 `target/`、`frontend/dist/`、`node_modules/` 之类构建产物带进仓库

## 5. Build 阶段必须走“小步闭环”，不要一句话做完整站

这次项目实战里，最有效的做法不是“把商城全部重写”，而是每次只打通一个真实链路。

推荐顺序：
1. 跑通登录链路
2. 跑通数据库建表和初始化
3. 跑通身份切换
4. 跑通商品发布和商品浏览
5. 跑通购物车下单
6. 跑通订单状态机
7. 最后再做分页、文案、品牌名等体验层优化

对应到这次项目，已经验证过的做法包括：
- 先看日志再修登录，而不是盲改前端
- 发现 Redis 报错就先查配置和端口，不把问题归咎于密码
- 发现数据库没表，就检查是否真的执行了 `schema.sql`
- 发现登录后 403，就查 token 落盘时机和后续请求顺序
- 发现用户能乱改订单状态，就把状态机和角色权限写回后端服务层

## 6. 本项目的真实踩坑记录，后续开发必须优先排查这些点

### 6.1 登录问题不要只看前端表单

这次登录问题真实根因包括：
- Redis 端口配置错误，导致登录后写 token 失败
- 前后端登录字段不一致，有的发 `username`，有的收 `email`
- token 还没落到本地，前端就开始请求用户详情，导致 403

结论：
- 登录问题必须同时检查日志、Redis、DTO、Axios 请求和 token 存储顺序

### 6.2 数据库问题先查“有没有真的初始化”

这次不是“SQL 没写”，而是：
- 启动时实际使用了根目录 `application.yml`
- 根配置原本没开启对应的 SQL 初始化

结论：
- 看到“数据库没表”，先查当前启动到底吃哪份配置，再查 `schema.sql` 是否执行

### 6.3 订单流程必须后端兜底

前端把下拉框删掉只能改善体验，真正防止乱改状态必须靠后端。

本项目后续开发必须坚持：
- 状态机写在后端 service 层
- 买家、卖家、管理员的操作权限分开
- 前端按钮只是引导，不是安全边界

### 6.4 身份切换不是 UI 文案问题，是权限链路问题

“买家 / 卖家切换”至少会影响：
- 用户资料接口返回
- 卖家中心可见性
- 商品发布权限
- 订单查询范围

不要只改一个按钮文本，就认为身份切换做完了。

## 7. 写 Prompt 时，必须像排障单一样具体

鱼皮文档强调 Prompt 要具体，这一点在本项目非常明显。

坏提示词：
- 修一下登录
- 优化订单

好提示词：
- 只修改登录链路。先排查 Spring Boot 日志里的 Redis 连接失败，再检查前端登录接口字段名和 token 落盘时机。不要改注册、商品、订单模块。
- 只修改订单状态逻辑。要求买家只能从待付款改为待发货、从待收货改为已完成；卖家只能把待发货改为待收货；前端去掉任意状态下拉框。

提示词越具体，返工越少。

## 8. 提交和回滚要比“继续让 AI 改”更重要

鱼皮文档建议每完成一个功能就提交一次，这个项目完全适用。

建议以后按功能提交：
- `fix: align login fields and token persistence`
- `fix: initialize mysql schema from root config`
- `feat: support buyer seller role switching`
- `fix: enforce order status transitions by role`
- `feat: paginate product list with page size 10`

这样一旦 AI 把新功能做坏，可以快速定位和回退，而不是在混乱工作区里继续补锅。

## 9. 本仓库后续推荐开发节奏

每做一个新需求，固定按下面流程：
1. 先补 `RESEARCH.md` 或在 issue 里写清目标和边界
2. 更新 `PRD.md`
3. 如果涉及数据结构、权限、配置，先更新技术设计文档
4. 明确本次只动哪些文件、哪些链路
5. 小步实现
6. 用日志、接口、页面三层一起验证
7. 通过后立即提交

## 10. 一句话原则

这个项目最适合的 Vibe Coding 方式不是“让 AI 自由发挥”，而是：

先把需求、边界、状态机、配置入口讲清楚，再让 AI 只改一个闭环，并且每跑通一段就提交一次。
